  • Domain Driven Design Domain-driven design (DDD) is an approach to software development that focuses on aligning business and technical concepts. The basic idea is that software should be based on the complexity of the domain in which it is modeled.

    Using a pet store as an example, the domain would include practical concepts like pets and customers. The domain model would include classes like Pet and Customer and methods like buyPet and sellPet.

    It could look something like this for a pet store:
    Pet
    Represents a pet in the pet store
    Customer
    Represents a customer in the pet store
    buyPet
    Method to purchase a pet
    sellPet
    Method to sell a pet

    DDD models serve as a common language between domain experts and developers. Taking the above example further, a domain expert might say, "I want to buy a pet," and the developer would know to call the buyPet method in the Pet class. This approach helps to ensure that the software accurately reflects the business domain and requirements.

    By modeling in software (example below in Python), we ensure that the code is a true reflection of the business domain.
    @dataclass
class Pet:
    name: str
    species: str
    price: float
    def __init__(self, name, species):
        self.name = name
        self.species = species
        self.price = price

@dataclass
class Customer:
    name: str
    pets: List[Pet]
    def __init__(self, name):
        self.name = name
        self.pets = []

@dataclass
class OrderItem:
    pet: Pet
    quantity: int

@dataclass
class Order:
    customer_name: str
    order_items: List[OrderItem]

def create_order(customer_name, order_items):
    return Order(customer_name, order_items)

def total_price(self):
    return sum(item.pet.price * item.quantity for item in self.order_items)

def buy_pet(customer: Customer, pet: Pet):
    customer.pets.append(pet)

def sell_pet(customer: Customer, pet: Pet):
    customer.pets.remove(pet)

  • TDD (and what I also recommend)
    What is TDD?

    You are not allowed to write any production code unless it is to make a failing unit test pass.

    TDD is an excellent concept with value, as shown in the previous section's example, but it is rarely utilized for an extended period.

    While I strongly advocate TDD and its potential benefits, real-world development teams face challenges in fully realizing its value. In a business environment where speed is crucial, TDD is often experimented with but eventually abandoned simply because it's hard to quantify the effort to value.

    Unit tests, while essential, may only sometimes be the best investment for test coverage. System and integration tests, due to their broader focus, are often more cost-effective and less likely to miss critical issues. Starting test efforts with E2E (end-to-end) tests, which are quickest to implement and can uncover unforeseen bugs, can provide a more comprehensive test coverage. Five well-designed E2E tests that validate a user's experience can instill more confidence in the team than a thousand passing unit tests of the nature (test_input_is_type_float64). Another way to phrase it is if I was given 100 "Quality Points" to test a feature, I might spend 80 on E2E tests, 10 on security, 8 on performance and load, and 2 on unit tests.

  • SOLID, MAN! Solid is a mnumonic acronym for a set of design principles intended to make code cleaner, and more flexible to future enhancements
    • S Single responsibility principle The idea behind the SRP is that every class, module, or function in a program should have one responsibility/purpose in a program. As a commonly used definition, "every class should have only one reason to change". For example, to follow a single responsibility principle, we could break the following: 
      class Pet:
    def bark(self):
        return 'Woof!'
    def meow(self):
        return 'Meow!'
    def mow_lawn(self):
        return 'Mow!'
      Into:
      class Dog:
    def bark(self):
        return 'Woof!'

class Cat:
    def meow(self):
        return 'Meow!'

class LawnMower:
    def mow_lawn(self):
        return 'Mow!'
    • O Open/Closed Principle The open-closed principle states that software entities should be open for extension, but closed for modification. An example of this is a method that can be extended to add new functionality: 
      class Dog(Pet):
    def perform_trick(self, trick):
        switch(trick):
            case: 'sit'
                return 'Sitting'
            case: 'roll over'
                return 'Rolling over'

      A better way to implement this would be to use a strategy pattern where you pass a trick object to the Dog class: 
          class Dog(Pet):
        def perform_trick(self, trick):
            return trick.perform()
      This allows the "trick" to be implemented in a separate class: 
      class Sit:
    def perform(self):
        return 'Sitting'

class RollOver:
    def perform(self):
        return 'Rolling over'


      This allows new "tricks" to be added without risk of breaking existing code.
    • L Liskov Substituion Principle The Liskov substitution principle simply implies that when an instance of a class is passed/extended to another class, the inheriting class should have a use case for all the properties and behavior of the inherited class.

      This just means that if you have a class that inherits from another class, it should be able to be used in place of the parent class without any unexpected behavior. For example, this is not good because a Cat does not bark: 
      class Pet:
    def bark(self):
        return 'Sound!'

class Dog(Pet):
    def bark(self):
        return 'Woof!'

class Cat(Pet):
    def bark(self):
        raise NotImplementedError('Cats do not bark')


      Instead, we should have a separate method for the Cat class: 
      class BarkingPet:
    def bark(self):
        return 'Woof or bark'

    • I Interface Segregation Principle The interface segregation principle states that the interface of a program should be split in a way that the user/client would only have access to the necessary methods related to their needs.

      In layman terms, this means we shouldn't have a big interface that forces a class to implement methods it doesn't need. Instead, we should have smaller, more specific interfaces that only contain the methods that are relevant to the class.

      In Python, an interface is not a formal concept but still works:

      A bad interface: 
      class Animal:
    def bark(self):
        return 'Woof!'
    def meow(self):
        return 'Meow!'


      The problem is that a class is forced to implement both methods. A better way to do this would be to split the interface into two separate interfaces: 
      class Barking:
    def bark(self):
        return 'Woof!'
class Meowing:
    def meow(self):
        return 'Meow!'
    • D Dependency Inversion Principle The dependency inversion principle states: High-level modules should not import anything from low-level modules. Both should depend on abstractions (e.g., interfaces) and abstractions should not depend on details.

      In practice, this means that classes should depend on interfaces rather than on concrete implementations. This allows for more flexibility, easier testing, and is more "future-proof".

      A common example of these is with databases, where you might have a class that interacts with a database. Instead of directly importing the database class, you would create an interface that the database class implements.

      If done correctly, this could allow you to swap underlying database technolegy like MariaDB for MS SQL Server or NoSQL without altering the code.

  • Big O Notation Refers to the worst-case scenario for an algorithm.
    Constant Time O(1)
    The execution time remains constant regardless of the input size. This is rare but can occur for simple operations like accessing an element in a list by index.
    Logarithmic Time O(log n)
    The execution time grows logarithmically with the input size (n). This is often seen in algorithms that involve searching sorted data structures like binary search trees.
    Linear Time O(n)
    The execution time increases proportionally with the input size. This is common in algorithms that iterate through the entire input data set once, like calculating the sum of elements in a list.
    Quadratic Time O(n^2)
    The execution time grows quadratically with the input size. This occurs in algorithms that involve nested loops iterating over the input data, leading to potentially many comparisons or calculations

    UML
    (Unified Modeling Language) is a standardized visual language used for modeling software systems.
    Class Diagram
    Class diagrams are a type of UML diagram that shows the structure of a system by defining the classes, attributes, and methods of the system.
    Sequence Diagram
    Sequence diagrams are a type of UML diagram that shows how objects interact in a time period. They are used to visualize the interactions between objects in a system.
    Activity Diagram
    Activity diagrams are a type of UML diagram that shows the flow of activities in a system. They are used to visualize the steps in a process or workflow.
    PlantUML
    is a tool that generates diagrams from a simple text description. It supports a variety of diagram types, including UML diagrams, but also goes beyond UML to offer other types like flowcharts, activity diagrams, and more.
